# サンプルアプリ構想 (App Concept)

## なぜサンプルアプリが必要か

本書のテーマはAIエージェントのObservabilityである。概念説明だけでは「何をどう計装するか」が抽象的に終わってしまうため、全章を貫く1つのサンプルエージェントアプリを題材にする。章が進むにつれてアプリに計装を積み増していき、最終的には「判断を可視化できるエージェント」が手元に残る構成とする。

## サンプルアプリ: `travel-helper`

ごく小さな旅行プラン作成エージェント。ユーザーから旅先の希望（都市名・日数・キーワード）を受け取り、LLMが判断しながら複数ツールを呼び、最終的におすすめ行程を返す。

### なぜこのテーマか

- **判断が複数発生する**: 「天気を調べるべきか」「レストラン検索をすべきか」等、LLMがツール選択を繰り返す。Observabilityで「なぜそう判断したか」を追う題材に向く
- **外部API呼び出しを模倣できる**: 実APIに依存せず、スタブツール（ダミーの天気API・レストラン検索）で再現可能。読者が鍵やクォータを用意せずに動かせる
- **小さく保てる**: コード量はPython 200〜300行規模に収めることができ、本書の本題から逸れない

## エージェントのふるまい（全体像）

```
ユーザー入力: { city: "京都", days: 2, keywords: ["寺", "和食"] }
          │
          ▼
  [travel-agent]
   ├─ plan_stage: キーワードから調査項目を決定（LLM呼び出し1回）
   ├─ gather_stage: 必要なツールを順次呼ぶ
   │     ├─ weather_tool(city, days)  ← スタブ実装
   │     ├─ places_tool(city, keyword) ← スタブ実装（キーワード毎に繰り返し）
   │     └─ restaurant_tool(city)     ← スタブ実装
   └─ synthesize_stage: 収集情報からおすすめ行程を生成（LLM呼び出し1回）
          │
          ▼
  ユーザーへの応答: テキスト形式の行程案
```

1つのユーザーリクエストで、LLM呼び出しが2回、ツール呼び出しが3〜5回発生する。

## LLMバックエンド

- OCI Generative AI Service をOpenAI SDK互換エンドポイントで利用
- 環境変数 `OCI_GENAI_ENDPOINT` `OCI_GENAI_API_KEY` `OCI_GENAI_MODEL` で接続先を切り替え可能にする
- 読者が用意できない場合のフォールバックとして、`MOCK_LLM=true` でダミーレスポンスを返す「Mockモード」を提供

## スタブツール

外部API依存を避けるため、以下はローカル実装とする:

- `weather_tool(city, days)`: 固定パターンの天気情報を返す
- `places_tool(city, keyword)`: 都市名 + キーワードから架空のスポット3件を返す
- `restaurant_tool(city)`: 都市名から架空のレストラン3件を返す

スタブ実装は意図的に100〜300msの遅延とランダムなエラー（5%程度）を含め、Observabilityで観測する対象として十分な情報が生成されるようにする。

## クリーンアップ

- 全リソースは `aio11y-book` namespaceに配置
- 各章のMakefileに `make clean` ターゲット（その章のリソースのみ削除）
- リポジトリルートの `make clean` で `aio11y-book` namespace全削除
- 共有Observabilityスタック（`observability` `langfuse` namespace）には一切変更を加えない

## スコープ外

- 認証・認可（エージェントAPIは認証なしで起動。検証環境前提）
- 永続化（DBは使わない。状態はメモリのみ）
- 複数ノードへのスケーリング
- 本番運用のSLO／アラート設計
