# 執筆ガイドライン (Writing Guidelines)

本書全章で統一する文体・表記・図表・コード例・引用のルールを定義する。全章の執筆とレビューはこのガイドラインに従う。

## 文体ルール

### 基本文体
- **文体**: 常体（である調）
- **一人称**: 使用しない（「本書では」「ここでは」「以下では」を使用）
- **読者への語りかけ**: 最小限（「〜してみよう」「〜してください」は避け、「〜する」「〜を行う」と記述する）
- **命令形**: 手順説明では「〜を実行する」「〜を記述する」と淡々と述べる

### 文の長さ
- **推奨**: 1文60字以内
- **上限**: 1文80字
- **対処**: 80字を超える場合は分割する

### 段落の長さ
- **推奨**: 1段落3-5文
- **上限**: 1段落7文
- **対処**: 長い場合は段落を分割するか、箇条書きに整形する

### 接続詞
- 多用を避ける。特に「また」「そして」の連続使用は禁止
- 因果関係は「そのため」「したがって」を優先（「なので」は口語のため禁止）
- 対比は「一方」「これに対し」を使用

### トーン
- Observabilityツール群の「設計判断の根拠」を淡々と記述する
- 「なぜそうなっているか」を説明する際も推量表現を避け、事実と出典で語る
- ツールやベンダーを価値評価しない（「優れた」「劣る」等は避け、特性を述べる）

## 表記ルール

### 技術用語
- **初出時**: 日本語（英語）の形式
  - 例: 計装（Instrumentation）、分散トレーシング（Distributed Tracing）
- **2回目以降**: 日本語のみ（例: 計装）
- **英語のまま扱う用語**: 固有名詞・略語・識別子・コード中の記号（例: OpenTelemetry、OTLP、Span、Trace、Attribute、TracerProvider）
- **略語**: 初出時に正式名称を併記（例: OTLP（OpenTelemetry Protocol））。ただし本書タイトルに含まれる OpenTelemetry 等の頻出語は章冒頭で1度のみ展開
- **統一指定のある用語**: `docs/glossary.md` の表記を優先する

### 数字
- **1桁**: アラビア数字（1つ、3層モデル）を基本とする。慣用句は漢数字（一方、一例）
- **2桁以上**: アラビア数字（10個、100件）
- **技術的な数値**: 常にアラビア数字（ポート4317、3シグナル、gRPC、HTTP/2）

### 記号
- **括弧**: 日本語中は全角括弧（）、英語・技術用語・コード識別子は半角()
- **コロン**: 説明の区切りには全角「：」。コード中の `:` は半角のまま
- **ピリオド・カンマ**: 日本語文中では「。」「、」

### 見出し
- **章タイトル**: `# 第N章 タイトル`（必要に応じ `― サブタイトル` を付す。ダーシは全角「―」）
- **節タイトル**: `## N.X タイトル`
- **小見出し**: `### タイトル`
- **箇条書き**: `-` を使用（`*` は使わない）。ネストは半角スペース2つで1段

## 図表ルール

### 図表番号
- **図**: 図X.Y（X=章番号、Y=章内連番）
- **表**: 表X.Y
- **リスト（コード引用）**: リストX.Y（サンプルコードからの引用に付ける）

### キャプション
- 図表・リストの直下に斜体で配置（例: `*図4.1: TraceとSpanの関係*`）
- 簡潔に内容を説明する（1文）
- リストにはソースファイルパスを含める（例: `**リスト13.2: `sample-app/ch13/agent.py`**`）

### 本文中の参照
- 「図4.1に示すように」「表4.2を参照」の形式
- 図表・リストは必ず本文中で参照する

### Mermaid記法
- ノード名は原則日本語（英語の固有名詞はそのまま）
- 矢印にラベルを付け、処理・データの流れを明示する
- シンプルに保つ（ノード数10個以内を推奨）
- 配色は以下の4色パレットに統一する:
  - 青系（`fill:#E3F2FD,stroke:#1976D2,color:#0D47A1`）: 計装層／アプリケーション
  - 紫系（`fill:#F3E5F5,stroke:#7B1FA2,color:#4A148C`）: 収集・転送層（Collector等）
  - 緑系（`fill:#E8F5E9,stroke:#388E3C,color:#1B5E20`）: 保存層（Prometheus／Loki／Tempo等）
  - 橙系（`fill:#FFF3E0,stroke:#F57C00,color:#E65100`）: 可視化・評価層（Grafana／Langfuse）
- テキスト図（ツリー表示等）はMermaidで表現できない場合のみ使用する

### 図表の配置原則
- 各節に最低1つの図表を配置する（`docs/book-plan.md`の方針）
- 章冒頭には、その章の位置付けを3層モデル上で示す「地図図」を可能な限り置く

## コード例ルール

### 言語
- Python（3.11以上）を基本とする。OTel／OpenLLMetry／Langfuse SDK呼び出しはPython
- Kubernetesマニフェスト、Collector設定、シェルスクリプトは適宜使用
- 疑似コードを用いる場合は冒頭コメントに `# 疑似コード` と明記

### 動作検証
- 原稿掲載コード・マニフェスト・設定ファイルは全て、本プロジェクトからアクセス可能なKubernetesクラスタ（OTel Collector／Prometheus／Loki／Tempo／Grafana／Langfuseデプロイ済み）で実機検証する
- 検証済みコードは `sample-app/chNN/` に配置し、原稿のコード例はそこからの引用とする
- 引用時は `**リストX.Y: `ファイルパス`**` 形式でキャプションを付ける

### 形式
- コードブロックには言語名を指定（```` ```python ```` ```` ```yaml ```` ```` ```bash ```` 等）
- コメントは必要最小限に絞り、「なぜ」を書く（「何を」はコードが語る）
- 完全な実装ではなく、概念理解に必要な最小限のコードに絞る（全量はサンプルコードリポジトリに委ねる）

### インデント
- Python: 4スペース
- YAML: 2スペース
- JSON: 2スペース

### 行長
- 80字を上限とする（ページ幅の制約）。超える場合は改行・変数抽出で対処

## 引用・参照ルール

### 情報ソースが必要な記述

以下の主張には必ず情報ソースを付ける:

- 歴史的経緯（「OpenTelemetryは2019年にOpenTracingとOpenCensusの統合として生まれた」等）
- 標準仕様の内容（W3C Trace Context、OTLP仕様、GenAI Semantic Conventions）
- プロジェクトの公式機能・仕様（Langfuseの機能、Prometheusのクエリ仕様等）
- 「〜が広く採用されている」「〜がデフォルトである」等の一般化した主張
- バージョンに依存する記述（「OpenLLMetry Xバージョン時点で〜に対応」）

### 情報ソース不要な記述
- 本書独自の定義・整理（3層モデルの分類等）
- 仕様に基づく論理的帰結
- 本書の他の章への参照
- コード例や実機検証の結果

### 情報ソースの書き方

本文中では脚注形式で記載する:

```markdown
OpenTelemetryは2019年にOpenTracingとOpenCensusの統合として生まれた[^1]。

[^1]: OpenTelemetry Project, "History", https://opentelemetry.io/docs/.
```

各章末に参考文献セクションを設ける:

```markdown
## 参考文献

- OpenTelemetry Project. "Specification." https://opentelemetry.io/docs/specs/
- Traceloop. "OpenLLMetry Documentation." https://www.traceloop.com/docs/openllmetry
```

### 外部資料の参照
- 本文中では脚注（`[^N]`）形式で参照し、URL・出典は脚注または章末参考文献にまとめる
- URLは可能な限り公式ドキュメントを参照し、ブログ記事は補助的に使う

### 章間の参照
- 「第X章で述べたように」「X.Y節を参照」の形式
- 前方参照（まだ説明していない内容への参照）は最小限にする
- 付録参照は「付録A参照」の形式

### コード・リソース引用
- サンプルコードからの引用はリスト番号とファイルパスを明示
- K8sマニフェスト・Collector設定も同じ形式でファイルパスを明示

## サンプルコードとクリーンアップ

- 全サンプルは専用namespace `aio11y-book` に配置する
- リポジトリルートに `make clean`（全削除）と `make clean-chNN`（章単位削除）を用意する
- 本文で初めてリソースを作る章では、末尾にクリーンアップ手順への導線を示す（例: 「この章で作成したリソースは `make clean-ch04` で削除できる」）

## 章末の定型構造

各章末に以下を含める:

1. **まとめ**: 箇条書きで5項目以内。この章で得た概念・スキルを整理
2. **理解度チェック**: 3〜5問（選択式または短答式）。解答は付録または章末脚注に配置
3. **参考文献**: 本章で参照した外部資料
4. **次章への接続**: 1〜2文で次章の位置付けを示す

## 禁止事項

- 推量表現: 「〜と思う」「〜でしょう」「〜かもしれない」（技術書では断定する。不確実な部分は「現時点ではexperimentalである」等と事実ベースで書く）
- 主観的な難易度表現: 「簡単に」「すぐに」「誰でも」
- 曖昧な時間表現: 「最近」「近年」「従来」（具体的な年を記載：「2024年以降」等）
- 読者を見下す表現: 「当然ですが」「言うまでもなく」「皆さんご存知の通り」
- 未定義の略語使用
- ベンダー・プロダクトの比較的価値評価（「Aの方が優れている」等）
- 再現不可能な記述: 「著者の環境では動いた」等、読者が追試できない記述
