## 節の構成

### 8.1 なぜGenAI向けの標準が必要なのか

**概要**: LLM呼び出しが従来のHTTPとは質的に異なる情報（プロンプト、レスポンス、トークン数、モデルパラメータ）を持つことを示し、ツール間の相互運用性が課題であったことを説明する。

**図表**:
- 図8.1: ツールごとに独自Attribute名を使っていた時代の問題点

**キーポイント**:
- 各ツールが独自命名で互換性なし
- ツール切り替え時にデータ移行が困難

### 8.2 GenAI Semantic Conventionsの概要

**概要**: OTelプロジェクト内のGenAI SIGが策定する標準Attribute名と、3シグナルへの対応を紹介する。

**図表**:
- 表8.1: 代表的な標準Attribute名一覧（`gen_ai.request.model`、`gen_ai.usage.input_tokens` 等）
- 図8.2: Traces／Metrics／Eventsへの対応

**キーポイント**:
- Attribute名が標準化されることで、ツール間の互換性が確保される
- 現状experimentalのためバージョン変更の可能性がある[要出典]

### 8.3 OpenLLMetryとOTel本体の合流

**概要**: TraceloopがOpenLLMetry計装コードをOTel本体に寄贈する動きを整理する。

**図表**:
- 図8.3: OpenLLMetry → OTel Contrib → OTel公式の移行経路イメージ

**キーポイント**:
- 将来的にOTel公式の計装パッケージとして提供される可能性
- 今OpenLLMetryを採用していても、将来の移行は比較的容易

### 8.4 実務への影響

**概要**: experimentalステータスの意味を理解した上で、今採用する場合の注意点を示す。

**図表**:
- 表8.2: 標準準拠のメリット・リスクと推奨対応

**キーポイント**:
- Attribute名は標準に揃える（移行容易性）
- 破壊的変更を受ける可能性は受容する設計
