# 第3章 OTelはなぜ生まれたか ― 標準化の背景 ― 仕様書

**章の目的**: OpenTelemetryがベンダーロックイン解消という動機のもとで誕生した経緯を理解し、「計装をバックエンドから切り離す」という設計判断の根拠を読者が自分で語れるようにする。

## 節の構成

### 3.1 ベンダーロックイン問題

**概要**: 分散トレーシング黎明期に各ベンダーが独自SDK・独自フォーマットを提供していたことで、アプリ計装がベンダー依存に陥った経緯を振り返る。

**図表**:
- 図3.1: ベンダーロックイン下での計装依存の模式図（アプリ→ベンダー固有SDK→ベンダー固有バックエンドの直結。バックエンド変更時に計装の書き直しが発生）

**キーポイント**:
- アプリ側コードがベンダーに密結合すると、バックエンド変更コストが高い
- 「計装」と「バックエンド」を分離したい動機が生まれた

### 3.2 OpenTracingとOpenCensus

**概要**: 2つの先行標準化プロジェクトの目的と範囲の違いを整理し、「2つの標準が並立する状態」がエコシステム分断を招いたことを示す。

**図表**:
- 表3.1: OpenTracingとOpenCensusの比較（発足年、主導、対象シグナル、問題点）

**キーポイント**:
- OpenTracingはトレース特化（CNCF）
- OpenCensusはトレース＋メトリクス（Google主導）
- 2つの標準の並存はアダプタ・ブリッジを必要とし、ユーザー体験を損ねた

### 3.3 OpenTelemetryの誕生

**概要**: 2019年にOpenTracingとOpenCensusが統合されOTelとなり、CNCFでIncubatingからGraduatedに昇格した流れを示す。Traces／Metrics／Logsの3シグナルをすべてカバーする標準としての位置付けを確認する。

**図表**:
- 図3.2: OTel誕生のタイムライン（2016年OpenTracing、2017年OpenCensus、2019年統合、2021年頃Tracesが安定、2023年以降Logsも安定）

**キーポイント**:
- 統合は2019年。ベンダーニュートラルを維持しつつ計装標準を1本化した
- 3シグナルを同じプロジェクト内で扱う
- 主要ベンダーが準拠する流れが加速した[要出典]

### 3.4 なぜこの背景が本書に関係するか

**概要**: OTelの設計思想（計装とバックエンドの分離、Collectorによる中継）が、本書のサンプル構成の出発点であることを明示する。また、OpenLLMetryがOTelの上に構築されている理由もこの文脈で理解する。

**図表**:
- 図3.3: 「計装・収集・保存」の分離が生む柔軟性（バックエンド差し替えが計装に影響しない）

**キーポイント**:
- OTel Collectorが中継点となることで、バックエンド変更がアプリに波及しない
- OpenLLMetryはOTelの計装標準の上で動くため、バックエンドを選ばない
- LLM向け計装のOTel本体への合流も、同じ論理で進行中（第8章で詳しく扱う）
