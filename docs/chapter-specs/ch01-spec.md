# 第1章 なぜAIエージェントにObservabilityが必要なのか ― 仕様書

## 章の概要

**章の目的**: 従来のマイクロサービスとAIエージェントシステムのObservability要件の違いを読者が自分の言葉で説明できるようになる。本書全体を貫く「システム挙動」と「LLMの判断品質」という2つの関心事を導入し、改善ループの具体像を提示する。

**目標ページ数**: 10p

**所属する部**: 第I部 全体像と知識地図

## 前提知識

この章を読むために必要な前提:
- 分散システム／マイクロサービスの基本概念（リクエスト・レスポンスの流れ、複数サービス間のやり取り）
- LLMを使ったアプリケーション開発の実務経験（OpenAI SDKやOCI GenAIの呼び出し経験）
- ログ・メトリクスという言葉の一般的な意味

本書の章としては最初の章のため、他章への依存はない。

## 節の構成

### 1.1 従来のマイクロサービスのObservability

**概要**: Observabilityという言葉が元々解決しようとしていた問題を振り返る。分散トレーシング・ログ・メトリクスの3点セットが、決定論的な処理フローに対してどう機能するかを確認する。

**図表**:
- 図1.1: 従来のマイクロサービスでのリクエスト追跡の典型例（ユーザー→APIゲートウェイ→サービスA→サービスB→DBの直列フロー、各ホップにログ・メトリクス・トレースが付く図）

**キーポイント**:
- 決定論的な処理フローでは「どこで何が壊れたか」を追跡すれば原因特定できる
- 従来のObservabilityは「システムの振る舞い」を観測する枠組みとして成熟している
- しかしAIエージェントではこの枠組みだけでは不足する

### 1.2 AIエージェントの根本的な違い

**概要**: 非決定論性・判断の内在化・エージェント間協調という3つの特性を導入し、従来のObservabilityでは捕捉しきれない側面を明らかにする。

**図表**:
- 図1.2: AIエージェントのリクエスト処理イメージ（LLM判断で経路が分岐する非決定論的フロー、動的なツール呼び出し、複数エージェント協調を含む図）

**キーポイント**:
- 同じ入力でもLLMの出力が変わるため、再現性を前提にした従来型デバッグが通用しにくい
- 「どのツールを呼ぶか」「どのエージェントにルーティングするか」はLLMが判断しており、システム側の静的な設計図では追えない
- 複数エージェントが連鎖的に動作し、1リクエストの全体像の把握難度が上がる

### 1.3 追うべき2つの関心事

**概要**: 本書の核となる整理として、Observability対象を「システムとして何が起きたか」と「LLMの出力品質はどうだったか」に分け、それぞれの代表的な問いと扱うべきデータを示す。

**図表**:
- 表1.1: 2つの関心事の比較（従来の関心事と新しい関心事、典型的な問い、扱うデータ、本書で対応するツール群）

**キーポイント**:
- 関心事A: システムとして何が起きたか（トレーシング中心。従来のObservabilityの延長）
- 関心事B: LLMの出力品質はどうだったか（プロンプト／レスポンス／評価スコア。従来のObservabilityにはなかった視点）
- どちらか片方では品質改善が回らない。両方が必要

### 1.4 改善ループの具体像

**概要**: 両方の関心事を使った実際の改善フローを3つのシナリオで示し、本書全体で読者が習得する能力のイメージを具体化する。

**図表**:
- 図1.3: 改善ループの模式図（検知→深掘り→仮説→修正→効果確認のサイクルを、Grafana／Tempo／Langfuseのアイコン付きで示す）

**キーポイント**:
- シナリオA: 「リクエスト全体が遅い」→ Grafana＋Tempoでボトルネック特定
- シナリオB: 「LLMが変な判断をした」→ Langfuseでプロンプト／レスポンス精査
- シナリオC: 「修正後の品質評価」→ Langfuseのスコア推移＋Grafanaのレイテンシ変化を横断確認

### 1.5 本書の読み方

**概要**: 本書が4部構成であり、第I部で全体像、第II部で概念、第III部でLLM固有のトピック、第IV部で実装に落とし込むことを示す。読者が各章をどう活用するかの道案内を行う。

**図表**:
- 図1.4: 本書の章構成マップ（`docs/book-architecture.md` の依存図を簡略化したもの）

**キーポイント**:
- 概念章（第II部）と実装章（第IV部）の対応関係
- サンプルアプリを章を追って発展させること
- 読者は必ずしも直線的に読む必要はなく、章間依存関係図を見て必要な章に飛べること

### 問題の方針
- 問題数: 3問
- 問題の種類:

  - 判断問題: 1問

### 問題案
1. 従来のマイクロサービスとAIエージェントシステムで、Observabilityの要件がどう違うかを、非決定論性の観点から説明せよ。（種類: 概念の確認）
2. 本書で扱う「2つの関心事」をそれぞれ1行で説明し、どのツール群で対応するかを述べよ。（種類: 概念の確認）
3. 「同じ質問に対してLLMが日によって違う答えを返す」という状況に対し、関心事Aと関心事Bのどちらの観点でObservabilityを強化すべきかを選び、理由を述べよ。（種類: 判断問題）
