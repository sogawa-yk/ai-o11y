# 第7章 自動計装と手動計装 ― 仕様書

## 章の概要

**章の目的**: 自動計装（モンキーパッチ）と手動計装の仕組みを理解し、「何を自動に任せ、何を手動で書くか」の判断基準を持つ。

**目標ページ数**: 12p

**所属する部**: 第II部 OTelの概念と仕組み

## 前提知識

- 第4章〜第6章（Span／3シグナル／Collector）
- Pythonのデコレータとモジュールインポートの仕組み

## 節の構成

### 7.1 自動計装の仕組み

**概要**: モンキーパッチでライブラリメソッドにフックを挿入する仕組みを説明する。`requests.get()` を呼ぶだけでHTTPリクエストがSpan化される流れを具体例で示す。

**図表**:
- 図7.1: モンキーパッチによる計装挿入のイメージ（オリジナル関数をラップして前後処理を挿入）

**キーポイント**:
- `opentelemetry-instrumentation-*` パッケージが対応ライブラリを提供
- 初期化1行で有効化できる
- 仕組み上、対応ライブラリがなければ動かない

### 7.2 手動計装の仕組み

**概要**: `with tracer.start_as_current_span()` で自分でSpan定義を書く方式を、ビジネスロジック固有の「判断ポイント」を記録する観点で再説明する。

**図表**:
- 図7.2: 手動計装が捕捉する「ビジネスロジック固有」のポイント

**キーポイント**:
- 完全な制御：Attribute命名、ネスト構造、Eventの付与
- 計装コードがビジネスロジックに混ざる

### 7.3 両者の比較

**概要**: 自動計装と手動計装のトレードオフを整理する。

**図表**:
- 表7.1: 自動計装／手動計装の比較（導入コスト、制御度、メンテコスト、対応範囲）

**キーポイント**:
- 自動は導入コスト最小・制御最小
- 手動は導入コスト高・制御最大

### 7.4 エージェント開発での使い分け

**概要**: AIエージェント開発での典型的な使い分けを示す。HTTP／LLM呼び出しは自動、判断ロジック・A2A通信・カスタムメトリクスは手動という方針を提示する。

**図表**:
- 図7.3: エージェントアプリの計装マップ（自動カバー範囲／手動カバー範囲を色分け）

**キーポイント**:
- LLM呼び出しはOpenLLMetry（自動）、判断ロジックは手動
- 両者は混在できる

### 7.5 両者を組み合わせたコード例

**概要**: 自動計装と手動計装が共存する状態のトレース構造を図と擬似コードで示す。実装例の動作確認は第13章・第14章で行う。

**図表**:

- リスト7.1: 自動計装＋手動計装の擬似コード

**キーポイント**:
- 両者のSpanは同じTraceContextを共有するため自然に親子関係を持つ
- Span名・Attributeの命名規則を揃えることで読みやすくなる

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 4問
- 種類: 概念の確認2問、判断問題2問

### 問題案
1. モンキーパッチとは何か、計装の文脈で説明せよ。（概念の確認）
2. 手動計装でしか記録できない情報の例を2つ挙げよ。（概念の確認）
3. 「PostgreSQLクライアントの呼び出しを計装したい」場合、自動／手動どちらを先に検討すべきか、理由と共に答えよ。（判断問題）
4. 「エージェントがどのツールを選んだかを記録したい」場合、どちらを使うべきか答えよ。（判断問題）
