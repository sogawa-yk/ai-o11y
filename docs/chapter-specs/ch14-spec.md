# 第14章 OpenLLMetryのセットアップと検証 ― 仕様書

## 章の概要

**章の目的**: OpenLLMetryをサンプルアプリに組み込み、OCI GenAI環境で動作検証を行う。手動計装との共存を実機で確認する。

**目標ページ数**: 12p

**所属する部**: 第IV部 実践 ― 収集と可視化

## 前提知識

- 第9章（OpenLLMetryの仕組み）、第10章（OCI GenAI検証観点）、第13章（手動計装完成版）

## 節の構成

### 14.1 セットアップ手順

**概要**: パッケージインストール、`Traceloop.init()` 設定、環境変数によるCollector接続・キャプチャ設定を実機で行う。

**図表**:
- リスト14.1: `requirements.txt` 差分（OpenLLMetry追加）
- リスト14.2: `sample-app/ch14/otel_setup.py` のTraceloop初期化部

**キーポイント**:
- OTel SDK初期化とTraceloop.init()の順序
- `TRACELOOP_TRACE_CONTENT=true` でプロンプト／レスポンスキャプチャを有効化
- 既存の手動計装と衝突しない構成

### 14.2 OCI GenAI環境での動作検証

**概要**: Chat Completions API／Responses APIそれぞれでOpenLLMetryの自動Span生成を確認する。

**図表**:
- 図14.1: 検証時の構成（travel-helper-ch14 → Gateway Collector → Tempo）
- 図14.2: Tempoで見た自動計装Spanの実スクリーンショット風図

**キーポイント**:
- `service.name=travel-helper-ch14` で検索
- `gen_ai.*` Attributeが記録されていることを確認
- Responses API内部のSpan構造を観察

### 14.3 手動計装との共存を確認する

**概要**: 第13章で書いた手動Spanと、OpenLLMetryの自動Spanが同じ木に収まることを実機で確認する。

**図表**:
- 図14.3: 手動Span＋自動Spanの混成木（実際のTempo表示）

**キーポイント**:
- `plan_stage` の下に `openai.chat` 等の自動Spanが自動的に配置される
- 親子関係はTracerProviderのContextで自動管理

### 14.4 プロンプト・レスポンスのキャプチャと注意点

**概要**: キャプチャ有効化で見える情報と、プライバシー・コンプライアンス面の注意を整理する。

**図表**:
- 表14.1: キャプチャON／OFFで記録される情報の違い

**キーポイント**:
- PII／機密情報が記録される可能性を認識する
- 本番ではキャプチャをOFFにする、あるいはCollectorのProcessorでマスクする選択肢がある

### 14.5 不足がある場合の手動補完

**概要**: 自動計装で捕捉できない情報を手動で補う場合のパターンを示す。

**図表**:
- リスト14.3: Responses API内部の判断理由を手動Attributeで補完する例

**キーポイント**:
- 将来OTel本体が対応したら差し替えやすい設計
- 手動補完部分を `otel_setup.py` とは別モジュールに分離

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 3問
- 種類: 概念の確認1問、判断問題1問、設計問題1問

### 問題案
1. OTel SDK初期化とTraceloop.init()の順序が重要な理由を述べよ。（概念の確認）
2. 本番環境でプロンプト／レスポンスキャプチャをONにするべきか、どう判断するか。（判断問題）
3. 自動計装で捕捉できないResponses API内部の判断理由を補完するため、どのような手動Attributeを設計するか。（設計問題）
