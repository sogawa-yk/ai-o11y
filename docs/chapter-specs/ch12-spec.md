# 第12章 OTelとLangfuseの役割分担とデータ送信経路 ― 仕様書

## 章の概要

**章の目的**: OTelとLangfuseの使い分けを明確にし、Langfuseへのデータ送信経路（OTLP／SDK／併用）の選択基準を持てるようになる。

**目標ページ数**: 14p

**所属する部**: 第III部 AIエージェント固有のObservability

## 前提知識

- 第9章（OpenLLMetry）、第11章（Langfuseの機能）

## 節の構成

### 12.1 役割分担の整理

**概要**: OTel+GrafanaとLangfuseが対応する関心事を、第1章の「2つの関心事」に対応付けて再確認する。

**図表**:
- 表12.1: OTel+Grafana／Langfuseの対応関心事・扱うデータ・代表的な問い

**キーポイント**:
- OTel+Grafana: システム全体の挙動
- Langfuse: LLM出力品質・プロンプト管理
- どちらか片方では改善ループが閉じない

### 12.2 trace_idによる紐付け

**概要**: OTelとLangfuseの両方に同じリクエストが記録される場合、trace_idで横断するデバッグフローを提示する。

**図表**:
- 図12.1: trace_id紐付けによる横断デバッグフロー（Grafanaでボトルネック検知→Langfuseで該当LLM呼び出しの品質確認）

**キーポイント**:
- OTel trace_idをLangfuseのmetadataに渡す
- 逆方向も可能（Langfuseで品質低下検知→Grafanaで関連メトリクス確認）

### 12.3 データ送信経路の選択

**概要**: LangfuseへのデータをOTLP経由／SDK経由／併用のどれで送るかという設計判断を、メリット・デメリット表と決定フローチャートで整理する。

**図表**:
- 図12.2: 3つの経路の構成図（A: OTLP経由、B: SDK直接、C: 併用）
- 表12.2: 3経路のメリット・デメリット
- 図12.3: 経路選択フローチャート

**キーポイント**:
- OTLP経由：計装一元化の利点、Langfuse固有機能は使えない
- SDK直接：Langfuse機能フル活用、計装にLangfuse依存混入
- 併用：各ツールの強みを活かす現実解

### 12.4 併用時の設計パターン

**概要**: 併用構成の代表パターンを `travel-helper` のサンプルに即して提示する。

**図表**:
- 図12.4: 併用パターンの典型構成（OTelでシステムトレース、Langfuse SDKで評価、trace_idで紐付け）
- リスト12.1: trace_id紐付けの実装イメージ（本章は設計、実装は第11章）

**キーポイント**:
- 計装は1箇所で行い、出力を2系統に流す設計
- Langfuse SDKへの依存は最小化（薄いラッパーで差し替え容易に）

### 12.5 判断基準のまとめ

**概要**: 具体的な要件シナリオを表にし、どの経路を選ぶかを指南する。

**図表**:
- 表12.3: 要件シナリオ別の経路選択ガイド

**キーポイント**:
- 評価・プロンプト管理が必要ならSDK必須
- システム観測の統一性重視ならOTel中心
- 現実的には併用が最も効果的
