# 第10章 OCI GenAI + OpenAI SDK環境での検証ポイント ― 仕様書

## 章の概要

**章の目的**: OCI Generative AI ServiceをOpenAI SDK経由で利用する環境で、OpenLLMetryがどこまで機能するかを検証する観点と判断基準を提示する。

**目標ページ数**: 12p

**所属する部**: 第III部 AIエージェント固有のObservability

## 前提知識

- 第9章（OpenLLMetryの仕組みと限界）
- OCI GenAIやOpenAI SDKの呼び出し経験

## 節の構成

### 10.1 OCI GenAI ServiceのOpenAI SDK互換性

**概要**: Responses APIとChat Completions APIの2種類のインターフェースがOpenAI SDK互換で提供されている点を整理する。

**図表**:
- 図10.1: OpenAI SDKがOCI GenAIを呼び出す構成（`base_url` 変更でバックエンドを切替）
- 表10.1: Chat Completions APIとResponses APIの違い

**キーポイント**:
- OpenAI SDKから `base_url` 変更だけで接続先変更可能
- Responses APIは内部でオーケストレーションを行う可能性がある

### 10.2 検証すべきポイント

**概要**: OpenAI SDK経由でのOpenLLMetry動作と、Responses API内部フローの捕捉範囲という2つの観点を示す。

**図表**:
- 図10.2: 2つの検証軸（SDK自動計装の有効性／Responses API内部の可視性）

**キーポイント**:
- Chat Completions APIは標準的に自動計装される想定
- Responses API内部のツール呼び出しがSpan化されるかは要検証

### 10.3 検証の手順

**概要**: 3段階の検証フロー（基本動作→Responses API→不足の特定）を提示する。

**図表**:
- 図10.3: 検証フローチャート（Step1→Step2→Step3）

**キーポイント**:
- Step 1: Chat Completions + OpenLLMetry基本動作
- Step 2: Responses APIでのSpan構造確認
- Step 3: 不足の特定と手動計装ポイントの洗い出し

### 10.4 判断基準

**概要**: 検証結果に応じて自動計装で十分／手動補完が必要の判断をどう下すかを整理する。

**図表**:
- 表10.2: 判断マトリクス（自動だけでOK／手動補完が必要なケース）

**キーポイント**:
- 自動で十分なら採用（メンテコスト低）
- 不足があれば手動計装で補完し、差し替え可能な設計にする

### 10.5 検証結果の記録と本書での前提

**概要**: 本書が想定するOCI GenAIの挙動（検証時点）を明示し、以降の章の前提とする。検証は執筆時の実機で行い、結果を本節に記録する。

**図表**:
- 表10.3: 本書検証時の結果一覧（Chat Completions / Responses API / トークン使用量 / プロンプトキャプチャ の各項目）

**キーポイント**:
- 結果は変わる可能性があるため「検証時点」と明記
- 読者の環境では再検証が必要

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 3問
- 種類: 概念の確認1問、判断問題2問

### 問題案
1. Chat Completions APIとResponses APIの違いを、Observabilityの観点でどう捉えるべきか述べよ。（概念の確認）
2. 自動計装の結果、Responses API内部が1つのSpanにしか見えなかった場合、次にどうすべきか。（判断問題）
3. 自動計装で十分な情報が取れるのに手動計装も書くことのデメリットは何か。（判断問題）
