# 第2章 ツール群の知識地図 ― 仕様書

## 章の概要

**章の目的**: 本書で扱うツール群を3層モデル（計装層／収集・転送層／保存・可視化層）で整理し、各ツールの役割と接続関係を読者が1枚の地図として把握できるようにする。

**目標ページ数**: 12p

**所属する部**: 第I部 全体像と知識地図

## 前提知識

- 第1章（2つの関心事と改善ループ）
- Kubernetes上でのデプロイ経験（Pod／Service／namespaceの基礎）

## 節の構成

### 2.1 3層モデルで整理する理由

**概要**: 個別ツールを列挙する前に、観測データの「生成」「転送」「活用」という3つの役割を層として分離し、そこに既存ツール群をマッピングする枠組みを提示する。

**図表**:
- 図2.1: 3層モデルの全体像（計装層・収集転送層・保存可視化層を縦積みにし、各層に配置されるツールを図示）

**キーポイント**:
- 3層はそれぞれ異なる関心事を扱う
- 層を越えて責務が混ざる設計はメンテナンス性を損なう
- 本書の章構成もこの3層に沿う

### 2.2 計装層のツール

**概要**: OTel SDK・OpenLLMetry・Langfuse SDKの3つを計装層として位置付け、それぞれが生成するデータと宛先を示す。

**図表**:
- 表2.1: 計装層のツール比較（生成データ／自動化度／対応範囲／送信先）

**キーポイント**:
- OTel SDKは汎用計装（手動）。OpenLLMetryはLLM SDK向けの自動計装
- Langfuse SDKはLLMの入出力と評価を記録する独自計装

### 2.3 収集・転送層のツール

**概要**: OTel Collectorがアプリとバックエンドを分離する中継点であることを示し、Receivers→Processors→Exportersのパイプライン構造を導入する。

**図表**:
- 図2.2: OTel Collectorのパイプライン概念図（3段構成、複数パイプラインの独立動作）

**キーポイント**:
- Collectorはアプリとバックエンドの疎結合化を担う
- 3シグナル（traces／metrics／logs）ごとに独立したパイプラインを持つ
- 詳細は第6章で扱う

### 2.4 保存・可視化層のツール

**概要**: Prometheus／Loki／Tempoがデータストア、Grafana／Langfuseが可視化UIであることを整理し、Langfuseがストア兼UIである点を明確にする。

**図表**:
- 表2.2: 保存・可視化層のツール比較（対象データ／クエリ言語／用途）
- 図2.3: データストアと可視化UIの接続関係

**キーポイント**:
- Grafanaは3ストアを横断するシステム観測のUI
- LangfuseはLLM出力品質のUI。独自のストアを内包する

### 2.5 データフロー全体図

**概要**: 計装→収集→保存→可視化の流れを1枚の図にまとめ、OTel経由の経路とLangfuse SDK経由の経路が並存することを示す。

**図表**:
- 図2.4: データフロー全体図（本書の正式なマップ。以降の章で参照される）

**キーポイント**:
- OTel経路とLangfuse経路は独立に動作し、trace_idで紐付けられる
- この図は本書の「北極星」であり、以降の各章はこの図のどこを詳しく見ているかを毎回示す

### 2.6 「どっちを見ればいい？」の判断基準

**概要**: 「システムが正常に動いているか」はGrafana、「LLMが賢く動いているか」はLangfuseという判断基準を、具体的な問いのリストで示す。

**図表**:
- 表2.3: 問いとツールの対応表（トラブルシュート状況ごとの第一選択ツール）

**キーポイント**:
- 判断基準は固有名詞（ツール名）で覚えるのではなく、関心事（第1章の2分類）から選ぶ
- 慣れてきたら両方を横断的に見る

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 3問
- 種類: 概念の確認2問、判断問題1問

### 問題案
1. 3層モデルのそれぞれの層の役割を、1行ずつで説明せよ。（概念の確認）
2. OpenLLMetryとLangfuse SDKの違いを、計装対象と送信先の観点から述べよ。（概念の確認）
3. 「LLMがコストを過剰に使っている」という問題に最初に向き合うとき、Grafana／Langfuseのどちらを開くべきか、理由とともに答えよ。（判断問題）
