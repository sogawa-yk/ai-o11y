## 節の構成

### 9.1 OpenLLMetryとは何か

**概要**: Traceloop社が開発するLLM SDK向け自動計装ライブラリ群としてのOpenLLMetryを紹介する。

**図表**:
- 図9.1: OpenLLMetryの位置付け（OTel Pythonの上に構築された計装パッケージ群）

**キーポイント**:
- OSS／40以上の計装パッケージ
- OTelと互換（生成Spanは通常のOTel Spanとして扱える）

### 9.2 セットアップと基本的な使い方

**概要**: `Traceloop.init()` 1行で自動計装が有効になる流れを、擬似コードで示す。

**キーポイント**:
- 初期化は1箇所
- 対応SDKを普通に呼ぶだけで裏でSpan生成

### 9.3 自動で記録されるデータ

**概要**: 生成されるSpan名、Attribute、プロンプト／レスポンスキャプチャのON/OFF設定を整理する。

**図表**:
- 表9.1: 自動記録される情報一覧（Span名、主要Attribute、キャプチャ条件）
- 図9.2: 典型的なLLM呼び出しSpanの中身（Tempo表示イメージ）

**キーポイント**:
- `gen_ai.request.model`、`gen_ai.usage.input_tokens` 等GenAI規約準拠
- プロンプト／レスポンスのキャプチャはデフォルトOFF、環境変数で有効化

### 9.4 OpenLLMetryの限界

**概要**: 対応SDK外では動かない点、判断理由は記録できない点、フレームワーク依存のバージョン追随コストを説明する。

**図表**:
- 表9.2: OpenLLMetryが捕捉できる／できない情報

**キーポイント**:
- OCI GenAI固有SDKの自動計装がない可能性
- 「判断の理由」は別途手動計装が必要
- SDKバージョンアップで計装が壊れる可能性

### 9.5 デコレータによるワークフロー定義

**概要**: `@workflow`、`@task`、`@agent` デコレータでSpan階層を宣言的に定義する機能を紹介する。

**図表**:

- リスト9.2: デコレータ使用の擬似コード

**キーポイント**:
- 手動の `with` 句と同等だが見通しが良い
- 既存関数に後付けしやすい

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 4問
- 種類: 概念の確認2問、判断問題2問

### 問題案
1. OpenLLMetryが自動で記録する代表的なAttributeを2つ挙げよ。（概念の確認）
2. OpenLLMetryが苦手とする観測対象を2つ挙げよ。（概念の確認）
